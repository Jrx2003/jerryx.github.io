## 设计云上电商系统

### 需求分析

```
功能需求:
- 商品展示、搜索
- 购物车、下单
- 支付、订单管理

非功能需求:
- 峰值 QPS: 10万+
- 可用性: 99.99%
- 延迟: P99 < 200ms
```

### 架构设计

```
┌─────────────────────────────────────────────────────┐
│                    接入层                            │
│   CDN ──→ WAF ──→ 高防IP ──→ 负载均衡(SLB)         │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                    网关层                            │
│   API Gateway ──→ 认证鉴权 ──→ 限流熔断            │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                    服务层                            │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │用户服务 │ │商品服务 │ │订单服务 │ │库存服务 │   │
│  │(多副本) │ │(多副本) │ │(多副本) │ │(多副本) │   │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘   │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                    数据层                            │
│  Redis ──→ MQ(RocketMQ) ──→ MySQL/分库分表        │
│  (缓存)    (异步/削峰)       (持久化)               │
└─────────────────────────────────────────────────────┘
```

### 关键技术点

```
1. 秒杀场景优化:
   - 库存预热到 Redis
   - 令牌桶限流
   - 异步下单
   - 消息队列削峰

2. 订单系统:
   - 分库分表 (按用户ID)
   - 读写分离
   - 最终一致性

3. 库存扣减:
   - Redis 预扣
   - 消息确认扣减
   - 定时对账补偿
```

## 设计视频直播平台

### 需求分析

```
功能需求:
- 主播推流
- 观众拉流
- 弹幕互动
- 录制回放

非功能需求:
- 低延迟 (<3s)
- 高并发 (百万在线)
- 卡顿率 < 1%
```

### 架构设计

```
推流端 → 边缘节点 → 中心节点 ──┬──→ 转码处理 ──→ 存储
                              │
                              └──→ CDN ──→ 播放端

关键组件:
- 边缘推流: 就近接入
- 中心调度: 智能选路
- 实时转码: 多码率适配
- CDN分发: 全球加速
```

### 关键技术

```
1. 直播协议选择:
   - RTMP: 推流主流
   - HLS: 播放兼容好，延迟高(10s+)
   - DASH: 自适应码率
   - WebRTC: 超低延迟(<1s)，连麦场景

2. 首屏优化:
   - GOP 缓存
   - 智能预加载
   - 播放器优化

3. 成本控制:
   - P2P 加速
   - 智能码率
   - 冷流合并
```

## 设计大数据分析平台

### 需求分析

```
数据来源:
- 业务日志 (TB级/天)
- 用户行为数据
- 第三方数据

分析需求:
- 实时报表
- 离线分析
- 机器学习
```

### 架构设计

```
┌─────────────────────────────────────────────────────┐
│                    数据采集                          │
│  Filebeat ──→ Kafka ──→ Flume ──→ 数据湖(S3/OSS)  │
└──────────────────────┬──────────────────────────────┘
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│  实时计算    │ │  离线计算    │ │  即席查询    │
│  Flink      │ │  Spark      │ │  Presto     │
│  (秒级)     │ │  (小时级)   │ │  (分钟级)   │
└──────┬──────┘ └──────┬──────┘ └──────┬──────┘
       │               │               │
       └───────────────┼───────────────┘
                       ▼
              ┌────────────────┐
              │   数据仓库      │
              │  (Hive/Doris)  │
              └────────────────┘
                       │
                       ▼
              ┌────────────────┐
              │   BI/可视化     │
              │  (Superset)    │
              └────────────────┘
```

### 分层设计

```
数据分层:

ODS (原始数据层):
- 原始日志，不做处理

DWD (明细数据层):
- 清洗、格式化
- 轻度聚合

DWS (汇总数据层):
- 按主题汇总
- 日/周/月统计

ADS (应用数据层):
- 面向业务应用
- 直接查询展示
```

## 设计多活架构

### 架构目标

```
同城双活:
- 两个可用区同时提供服务
- RPO ≈ 0, RTO < 分钟级

异地多活:
- 多个城市同时服务
- 就近访问
- 故障自动切换
```

### 同城双活

```
┌──────────────┐          ┌──────────────┐
│   可用区 A   │◄────────►│   可用区 B   │
│  ┌────────┐  │          │  ┌────────┐  │
│  │ 应用层 │  │          │  │ 应用层 │  │
│  └────────┘  │          │  └────────┘  │
│  ┌────────┐  │          │  ┌────────┐  │
│  │ 数据层 │◄─┼──同步────┼─►│ 数据层 │  │
│  └────────┘  │          │  └────────┘  │
└──────────────┘          └──────────────┘

技术要点:
- 数据库主主同步
- 应用无状态
- 流量调度 (DNS/SLB)
- 数据冲突解决
```

### 异地多活

```
单元化架构:

用户请求 ──► 流量网关 ──► 单元分配
                               │
       ┌───────────────────────┼───────────────────────┐
       ▼                       ▼                       ▼
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│   北京单元   │       │   上海单元   │       │   深圳单元   │
│  (华北用户)  │       │  (华东用户)  │       │  (华南用户)  │
│  ┌────────┐  │       │  ┌────────┐  │       │  ┌────────┐  │
│  │ 应用   │  │       │  │ 应用   │  │       │  │ 应用   │  │
│  │ 数据   │  │       │  │ 数据   │  │       │  │ 数据   │  │
│  └────────┘  │       │  └────────┘  │       │  └────────┘  │
└──────────────┘       └──────────────┘       └──────────────┘

关键点:
- 用户分区 (按地域/用户ID)
- 数据分区同步
- 全局配置中心
```

- CDN分发: 全球加速
```

### 关键技术

```
1. 直播协议选择:
   - RTMP: 推流主流
   - HLS: 播放兼容好，延迟高(10s+)
   - DASH: 自适应码率
   - WebRTC: 超低延迟(<1s)，连麦场景

2. 首屏优化:
   - GOP 缓存
   - 智能预加载
   - 播放器优化

3. 成本控制:
   - P2P 加速
   - 智能码率
   - 冷流合并
```

## 设计多活架构

### 架构目标

```
同城双活:
- 两个可用区同时提供服务
- RPO ≈ 0, RTO < 分钟级

异地多活:
- 多个城市同时服务
- 就近访问
- 故障自动切换
```

### 同城双活

```
┌──────────────┐          ┌──────────────┐

技术要点:
- 数据库主主同步
