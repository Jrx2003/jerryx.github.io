微服务概述

单体 vs 微服务

单体架构:
┌─────────────────────────────────┐
│          单一代码库              │
│  ┌─────┬─────┬─────┬─────────┐  │
│  │用户 │订单 │支付 │库存    │  │
│  └─────┴─────┴─────┴─────────┘  │
│         共享数据库               │
└─────────────────────────────────┘

微服务架构:
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│ 用户服务 │ │ 订单服务 │ │ 支付服务 │ │ 库存服务 │
│ 用户DB  │ │ 订单DB  │ │ 支付DB  │ │ 库存DB  │
└────────┘ └────────┘ └────────┘ └────────┘
     ↑_________↑_________↑_________↑
              API Gateway

优势与挑战

优势挑战
独立部署分布式复杂度
技术异构数据一致性
团队自治运维成本高
弹性伸缩网络延迟

拆分原则

拆分策略

按业务能力拆分:
- 用户服务
- 商品服务
- 订单服务
- 支付服务

按领域驱动设计 (DDD):
- 限界上下文作为边界
- 聚合根定义服务
- 领域事件通信

服务粒度

黄金法则:
- 两个披萨团队能维护 (6-12人)
- 能独立部署
- 有明确边界

避免:
- 过细 → 分布式单体
- 过粗 → 单体副作用

通信方式

同步通信

REST API:
GET /users/{id}
POST /orders
PUT /payments/{id}

GraphQL:
query {
  user(id: 1) {
    name
    orders { total }
  }
}

gRPC:
- Protocol Buffers
- HTTP/2
- 流式支持

异步通信

# 消息队列解耦
class OrderService:
    def create_order(self, order_data):
        # 1. 保存订单
        order = self.db.save(order_data)

        # 2. 发送事件
        self.event_bus.publish("OrderCreated", {
            "order_id": order.id,
            "amount": order.amount
        })

        return order

# 支付服务监听事件
class PaymentService:
    def on_order_created(self, event):
        # 异步处理支付
        self.process_payment(event.order_id, event.amount)

数据管理

数据库设计

原则: 每个服务有自己的数据库

服务A ──→ DB A
服务B ──→ DB B
服务C ──→ DB C

禁止直接访问其他服务的数据库!

分布式事务

Saga 模式:
┌─────────┐    ┌─────────┐    ┌─────────┐
│ 创建订单 │ → │ 扣减库存 │ → │ 扣款    │
└─────────┘    └─────────┘    └─────────┘
      ↓             ↓              ↓
   成功:继续    失败:补偿        失败:补偿

补偿操作:
- 取消订单
- 恢复库存
- 退款

面试要点

Q1: 微服务拆分的原则？

A:

  • 单一职责
  • 业务边界清晰
  • 独立部署
  • 数据独立

Q2: 如何处理分布式事务？

A:

  • Saga 模式 (最终一致性)
  • TCC (Try-Confirm-Cancel)
  • 本地消息表
  • 最大努力通知

Q3: 服务间通信选择？

A:

  • 同步: REST/gRPC (实时性要求高)
  • 异步: 消息队列 (解耦、削峰)
