## Serverless 概述

### 什么是 Serverless

```
Serverless = 无服务器架构

核心特征:
- 无需管理服务器
- 自动扩缩容
- 按实际调用付费
- 事件驱动

提供商管理:
  服务器、网络、存储、虚拟化、容器、运行时

开发者关注:
  代码、业务逻辑
```

### 两种形态

| 类型 | 代表 | 适用场景 |
|------|------|----------|
| FaaS | AWS Lambda, 函数计算 | 事件处理、API |
| BaaS | Firebase, AppSync | 后端服务集成 |

---

## 架构演进

### 演进历程

```
传统架构 (IaaS):
App → VM → OS → Middleware → Runtime

容器化 (CaaS):
App → Container → Container Runtime → VM → OS

Serverless (FaaS):
App → Function → Platform

每个阶段抽象层次更高，运维负担更小
```

### Serverless 架构图

```
事件源 → API Gateway → 函数平台 → 后端服务
              ↓
         ┌─────────┐
         │ 函数1   │
         │ 函数2   │
         │ 函数3   │
         └─────────┘
              ↓
         数据库/存储/队列
```

---

## 技术实现

### 冷启动优化

```
冷启动过程:
1. 下载代码 (从对象存储)
2. 启动容器
3. 初始化运行时
4. 执行初始化代码
5. 处理请求

优化策略:
- 预留实例 (Provisioned Concurrency)
- 单实例多并发
- 精简依赖包
- 延迟加载
```

### 状态管理

```
无状态设计:
函数不保存状态，每次调用独立

状态持久化:
┌────────┐    ┌────────────┐    ┌────────┐
│ 函数   │ → │ 外部存储   │ → │ 数据库 │
└────────┘    │ (Redis/S3) │    └────────┘
              └────────────┘
```

---

## 应用场景

### 适用场景

```
✅ 适合:
- Web/API 后端
- 定时任务 (Cron)
- 文件处理 (上传触发)
- 数据 ETL
- IoT 数据处理
- 聊天机器人

❌ 不适合:
- 长时间运行任务 (>15分钟)
- 需要低延迟 (<100ms)
- 大规模计算
- 遗留系统迁移
```

---

## 面试要点

**Q1: Serverless 的优势和局限？**

A:
- 优势: 零运维、自动扩缩、按需付费、快速上线
- 局限: 冷启动延迟、执行时长限制、调试困难、供应商锁定

**Q2: 如何解决冷启动问题？**

A:
- 预留实例保持热状态
- 单实例多并发复用容器
- 优化代码包大小
- 使用专用 VPC 减少网络初始化

**Q3: Serverless 与传统架构对比？**

A:
- 成本: 流量低时便宜，流量高时可能更贵
- 运维: Serverless 大幅减少运维工作
- 性能: 传统架构更可预测
