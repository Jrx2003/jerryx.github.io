冷启动问题

什么是冷启动

冷启动: 函数首次调用或闲置后重新启动

延迟组成:
┌──────────┬──────────────────────────────┐
│ 阶段     │ 耗时                         │
├──────────┼──────────────────────────────┤
│ 代码下载 │ 100ms - 数秒 (取决于包大小)  │
│ 容器启动 │ 50-200ms                     │
│ 运行时   │ 100-500ms (Java最慢)         │
│ 初始化   │ 取决于代码 (数据库连接等)    │
│ 总计     │ 数百ms 到数秒                │
└──────────┴──────────────────────────────┘

影响因子

冷启动 vs 热启动:

冷启动: 0ms ──→ 2000ms ──→ 执行
热启动: 0ms ──→ 10ms ──→ 执行

对用户体验影响巨大!

启动阶段分析

详细流程

1. 调度决策
   └── 选择合适的计算节点

2. 代码准备
   ├── 从对象存储下载代码包
   ├── 解压到本地磁盘
   └── 加载依赖

3. 运行时初始化
   ├── 启动运行时进程
   ├── 加载运行时库
   └── 编译/解释代码

4. 应用初始化
   ├── 建立数据库连接
   ├── 加载配置
   └── 预热缓存

5. 处理请求

优化策略

1. 代码优化

# ❌ 不好: 大量依赖
import numpy
import pandas
import tensorflow  # 但只用了一次

# ✅ 好: 按需导入
def handler(event, context):
    if need_ml:
        import tensorflow  # 延迟导入

2. 连接复用

# ❌ 不好: 每次调用创建连接
def handler(event):
    db = create_connection()  # 耗时!
    result = db.query(...)
    return result

# ✅ 好: 全局复用
_db = None

def get_db():
    global _db
    if _db is None:
        _db = create_connection()
    return _db

def handler(event):
    result = get_db().query(...)
    return result

3. 预留实例

# AWS Lambda Provisioned Concurrency
ProvisionedConcurrencyConfig:
  ProvisionedConcurrentExecutions: 100

效果:
- 保持指定数量的实例热启动
- 适合可预测流量的场景
- 成本较高但延迟最低

4. 单实例多并发

# AWS Lambda 并发配置
# 一个实例可同时处理多个请求

好处:
- 实例复用率提高
- 冷启动次数减少
- 资源利用率提升

注意:
- 需要确保代码线程安全
- 共享连接要正确处理并发

SnapStart (AWS)

原理:
1. 发布版本时执行初始化
2. 创建内存快照 (Firecracker VM)
3. 调用时恢复快照

效果:
- Java 冷启动从数秒降至100-200ms
- 适用于初始化重的运行时

面试要点

Q1: 冷启动的主要原因？

A:

  • 代码下载和加载
  • 运行时初始化
  • 应用依赖初始化
  • 网络连接建立

Q2: 实际项目中的冷启动优化经验？

A:

  • 精简依赖，延迟导入
  • 数据库连接池全局复用
  • 使用预留实例
  • 启用单实例多并发
  • 选择合适的运行时 (Python/Node 比 Java 快)

Q3: 预留实例 vs 按需实例的成本权衡？

A:

  • 预留实例: 固定成本，适合稳定流量
  • 按需实例: 可变成本，适合波动流量
  • 折中: 部分预留 + 按需弹性
