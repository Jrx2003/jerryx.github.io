## 容器服务类型

### 服务分类

```
容器服务 spectrum:

自托管                    全托管
├──────────┬──────────┬──────────┬──────────┤
│ 自建 K8s │ 托管 K8s │ Serverless│ PaaS     │
│          │ (EKS)    │ (Fargate) │ (App    │
│          │          │           │ Engine)  │
├──────────┼──────────┼──────────┼──────────┤
│ 最高控制  │ 平衡     │ 简化运维  │ 最高效率 │
│ 最高运维  │ 主流选择 │ 成本优化  │ 功能受限 │
└──────────┴──────────┴──────────┴──────────┘
```

### 类型详解

| 类型 | 代表 | 适用场景 |
|------|------|----------|
| 自建 K8s | kubeadm | 有特殊定制需求 |
| 托管 K8s | EKS/ACK/GKE | 大多数场景 |
| Serverless | Fargate/ECI | 突发流量、批处理 |
| 边缘容器 | KubeEdge | IoT、边缘计算 |

## 托管 K8s 对比

### 三大托管 K8s

```
┌──────────────┬────────────────┬────────────────┬────────────────┐
│ 特性         │ Amazon EKS     │ Azure AKS      │ Google GKE     │
├──────────────┼────────────────┼────────────────┼────────────────┤
│ 控制平面     │ 托管/$72月     │ 免费           │ 免费           │
│ 节点管理     │ 托管/自管      │ 托管/自管      │ 托管/自管      │
│ 自动升级     │ 支持           │ 支持           │ 最成熟         │
│ 服务网格     │ 集成 Istio     │ 集成 OSM       │ Anthos         │
│ 特色功能     │ Fargate 集成   │ Windows 容器   │ Autopilot      │
└──────────────┴────────────────┴────────────────┴────────────────┘
```

### 阿里云 ACK

```
ACK 类型:
- 托管版: 托管 Master，需自管 Worker
- 专有版: 全托管，适合大规模
- Serverless: 无节点概念，按 Pod 计费

优势:
- 与阿里云产品深度集成
- 国内网络优化
- 合规认证齐全
```

## Serverless 容器

### 对比分析

```
传统 K8s vs Serverless:

传统 K8s:
┌─────────────────────────────────────┐
│ 预留节点 (固定成本)                  │
│ ┌─────┐ ┌─────┐ ┌─────┐            │
│ │Pod  │ │Pod  │ │     │ ← 资源碎片 │
│ └─────┘ └─────┘ └─────┘            │
└─────────────────────────────────────┘

Serverless:
┌─────────────────────────────────────┐
│ 按需分配                             │
│      ┌─────┐   ┌─────┐   ┌─────┐   │
│      │Pod  │   │Pod  │   │Pod  │   │
│      └─────┘   └─────┘   └─────┘   │
│ 无节点概念，秒级启动，按秒计费       │
└─────────────────────────────────────┘
```

### 代表产品

| 产品 | 厂商 | 特点 |
|------|------|------|
| Fargate | AWS | 与 EKS/ECS 集成 |
| ECI | 阿里云 | 与 ACK 无缝集成 |
| CCI | 华为云 | 安全容器 |
| Cloud Run | GCP | Knative 基础 |

## 选型决策树

### 决策流程

```
开始
  │
  ▼
是否有 K8s 运维经验? ──否──→ 全托管 Serverless
  │是
  ▼
是否需要特殊 K8s 定制? ──是──→ 自建 K8s
  │否
  ▼
流量是否稳定可预测? ──是──→ 托管 K8s + 预留实例
  │否
  ▼
混合流量模式? ──是──→ 托管 K8s + Serverless 混合
  │
  ▼
托管 K8s (标准选择)
```

### 成本考虑

```
成本公式:

传统节点:
总成本 = 节点数 × 单价 × 时间
      (即使不用也付费)

Serverless:
总成本 = Pod 实际运行时间 × 单价
      (真正按用量付费)

盈亏平衡点:
- 资源利用率 > 60%: 预留更划算
- 资源利用率 < 60%: Serverless 更划算
```

## 面试要点

**Q1: 托管 K8s 和自建 K8s 怎么选？**

A:
- 团队规模小/缺乏运维经验: 托管
- 有特殊定制需求: 自建
- 大多数场景: 托管是更好选择

**Q2: Serverless 容器的限制？**

A:
- 启动延迟 (秒级)
- 不支持 DaemonSet
- 网络限制 (无法 hostNetwork)
- 调试困难

**Q3: 容器服务的安全考虑？**

A:
- 镜像扫描
- 运行时安全
- 网络隔离
- Secret 管理
- RBAC 权限控制
