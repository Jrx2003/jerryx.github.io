DeepEval 与 OpenAI Evals 对照

DeepEval 与 OpenAI Evals 对照

更新日期：2026-03-12 来源：docs/work_logs/LLM_JUDGE_EVALS_RESEARCH.md

目的

本项目已经有 CAD LLM Judge 评估链路，本文的目标不是"重做一套"，而是：

  1. 盘点当前已实现能力。
  2. 对照 deepeval 与 openai-evals 提炼可增量补强点。
  3. 给出分阶段落地建议。

已实现能力盘点

主题当前项目状态结论
JSON 结构化输出 + schema 校验已有（Pydantic 严格校验）✅ 已实现
Judge 输出无效时自动重试已有（schema_fix 二次重试）✅ 已实现
规则门限（hard gate）已有（missing/empty STEP 限分）✅ 已实现

更丰富 deterministic gate（STEP 可解析性等）目前仅 2 个 gate⚠️ 可增强

DeepEval 可借鉴点

1. 将"标准描述"与"评估步骤"分离

DeepEval 的 GEval 支持：

  • criteria（自然语言标准）
  • 或显式 evaluation_steps

对本项目的价值：

  • 当前 rubric 有 anchors，但缺显式步骤化检查序列
  • 可在 rubric 中加入 evaluation_steps，提升判分一致性与可解释性

2. DAG 思路：先规则、后语义

项目已具备雏形（RuleEvaluator + LLMJudge）。可继续扩展 deterministic gate：

  • STEP 基本可解析性检查
  • 实体数量/空几何异常检查
  • 明确几何异常信号后再进入语义评分

3. logprob 平滑（可选）

DeepEval 在支持 logprob 时可做分数平滑。该项可放后续，不建议作为当前优先级。

OpenAI Evals 可借鉴点

1. 离散 choice 判分提高稳定性

OpenAI Evals 常用 choice_strings + choice_scores。

对本项目建议：

  • 每个维度先给离散等级（如 A/B/C/D）
  • 服务端映射为分数（如 A=1.0, B=0.7, C=0.4, D=0.0）
  • 可保留模型给的 float 作为诊断字段，但主判分采用映射值，减少模型漂移影响

2. Meta-eval 校准回路

OpenAI Evals 推荐用人工标签数据集评估 judge 本身质量。

对本项目现状：

  • 已有 confidence 字段与配置
  • 但未看到独立"judge 校准集 + metascore"回路

建议补齐：

  • 构建小规模标注集（50-200 样本）
  • 跑每维度准确率与整体 metascore
  • 作为 prompt/rubric/model 变更的准入门槛

3. 评测集版本治理

沿用 <name>.<split>.<version> 思路管理 CAD judge 评测集，便于回归比较与复现。

分阶段落地建议

P0（先补闭环）

  1. 接通 confidence 触发重判（可同模型二次、或备用模型复判）
  2. 建立 judge 校准数据集与 metascore 脚本（放在 tests/evals/）
  3. 增加评估报表里的聚合指标：invalid 输出率、低置信度占比、重判命中率

P1（提升稳定性）

  1. 引入每维度离散 choice + 服务端 score 映射
  2. 扩展 deterministic gates（STEP 可解析性、基础几何异常信号）

P2（进阶）

  1. 更 DAG 化的评估路径（规则节点 + 语义节点）
  2. 在 provider 支持前提下探索 logprob 平滑
