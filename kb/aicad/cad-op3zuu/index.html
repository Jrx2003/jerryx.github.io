CAD Agent 迭代式演进方案

CAD Agent 迭代式演进方案

更新日期：2026-03-12 来源：docs/work_logs/CAD_AGENT_ITERATION.md

背景

当前系统使用单一的"意图解析 + 规划器"模式，类似于传统的一次性工作流。为了实现类似 AI/LLM 的迭代式演进，需要引入更智能的多 Agent 协作和持续优化机制。

核心改造方向

1. 引入多 Agent 角色（而非单一意图解析器）

当前问题

# 当前：单一 intent.py 处理所有用户输入
intent.parse(state) → 返回单一意图

改造方案：角色专业化 Agent

Agent 架构：

User Input
    ↓
Router (角色路由)
    ├─ RequirementAnalyzer (需求分析)
    ├─ CadGenerationAgent (CAD 生成)
    ├─ ClarificationAgent (澄清)
    └─ GreetingAgent (问候)

核心 Agent 职责：

Agent职责
RequirementAnalyzer智能识别用户真实意图（创建、修改、评估、澄清）
TaskPlanner将复杂需求分解为可执行的子任务序列
CadGenerationAgent生成 CAD 模型的迭代式演进
FeasibilityChecker多维度评估可行性（几何、制造、材料）
KnowledgeQuery查询材料数据库、标准、设计规范
IterationAgent基于反馈持续改进
EvaluationAgent自动化评分和质量检查
ParametricOptimizer基于设计规范优化几何参数

2. 迭代式 CAD 生成（分步优化）

核心概念

迭代式生成：类似于 AI 模型的生成过程，从粗略到精细逐步改进

示例流程：

用户需求："iPhone 15 手机壳"

第 1 轮：快速生成粗略版
├─ 意图：基本外形 → 快速反馈
└─ 评估：可行性 ✓

第 2 轮：基于反馈优化
├─ 意图：优化尺寸 → 精确参数
├─ 结构：添加细节 → 内部结构
└─ 评估：质量评分

第 3 轮：最终版
├─ 意图：完整模型 → 满足需求
├─ 结构：所有特征 → 可用 CAD 文件
└─ 评估：通过评分

实现架构

# services/main_agent/src/graph/state/
class IterationState(BaseModel):
    """迭代状态管理"""
    generation_chain: list[Generation]  # 生成链
    evaluations: list[Evaluation]       # 评估链
    current_iteration: int = 0         # 当前轮次
    best_score: float | None = None   # 最佳评分

3. 任务拆解与分步执行

当前问题

# 当前：planner 直接创建一个 job，无法跟踪子任务
planner.plan_cad() → job.create()

改造方案：任务图执行引擎

任务图结构：

TaskGraph
    ├─ TaskNode (任务节点)
    │   ├─ ExecutionAgent (执行)
    │   │   ├─ ValidationAgent (验证)
    │   │   └─ FeedbackCollector (反馈收集)
    │   └─ DependencyResolver (依赖解析)
    └─ StateManager (状态管理)

关键概念：

  • 子任务可独立追踪：每个几何特征创建、验证都是独立状态
  • 依赖关系管理：结构创建完成后才能开始特征创建
  • 反馈闭环：验证失败可以触发任务重试或修改

4. 智能评估与参数优化

改造方案：多维度评估系统

评估引擎架构：

EvaluationEngine
    ↓
    ├─ GeometryValidator (几何验证)
    │   ├─ MaterialChecker (材料检查)
    │   ├─ ManufacturabilityValidator (制造工艺)
    │   └─ ConstraintValidator (约束验证)
    ↓
    ParametricOptimizer (参数优化)
            ↓
        ├─ DesignStandardsLoader (标准库)
        │   └─ OptimizationRulesEngine (规则引擎)
            ↓
        RecommendationEngine (推荐引擎)
                ↓
        EvaluationTracker (评分跟踪)

评估维度：

维度指标说明
几何is_valid_geometry()是否可制造
材料is_material_available()库中是否有此材料
制造is_manufacturable()是否适合标准工艺
约束check_design_rules()是否违反设计规范
成本estimate_cost()预估制造成本
质量assess_quality_score()综合质量评分

5. CAD 模型编辑与版本管理

核心特性

  • 参数可追溯：每个设计决策都有来源和理由
  • 版本对比：可以查看不同版本的差异
  • 用户干预：允许用户手动调整参数
  • 迭代可视化：显示从 V1 到 Vn 的变化路径

版本管理架构：

ParametricModel
    ├─ Parameters (可编辑参数)
    │   ├─ GeometricParams (几何参数)
    │   ├─ FeatureParams (特征参数)
    │   └─ Constraints (约束参数)
    └─ VersionControl (版本控制)
        ↓
    ModelEditor
            ├─ ParameterEditor (参数编辑)
            ├─ FeatureEditor (特征编辑)
            └─ VersionManager (版本管理)

实施路线图

Phase 1: 多 Agent 架构重构（2-4 周）

优先级 P0

  • 创建新的 agents 目录结构
  • 实现 Agent 基类
  • 实现 RequirementAnalyzer
  • 实现 CadGenerationAgent
  • 实现 TaskPlanner
  • 更新 LangGraph 状态管理

Phase 2: 迭代式生成（3-4 周）

优先级 P1

  • 实现两阶段生成流程
  • 引入反馈循环机制
  • 扩展数据库（材料、规范）
  • 更新 API 路由

Phase 3: 任务图执行引擎（4-6 周）

优先级 P1

  • 实现任务图系统
  • 实现子任务数据模型
  • 更新 Job API 路由

Phase 4: 智能评估系统（5-7 周）

优先级 P1

  • 实现自动化评估系统
  • 扩展 API（迭代、反馈）
  • 前端可视化更新

预期效果

功能当前状态改造后效果
智能意图识别单一 intent 分类多 Agent 角色，准确率 >90%
迭代优化无反馈机制自动反馈循环，持续改进
任务跟踪单 job 状态任务图，可独立追踪子任务
评估系统手动评估自动多维度评估，客观评分
版本管理无版本控制参数化模型，版本对比
用户反馈无收集渠道结构化反馈系统，可分析趋势

风险与注意事项

技术风险

  1. 复杂性增加：多 Agent 架构需要更复杂的状态管理
  2. 性能开销：任务图执行比单一 planner 慢
  3. 数据库依赖：材料库和规范库需要额外的存储和查询
  4. LLM 成本：多轮迭代会显著增加 token 使用

缓解策略

  1. 增量实施

    • 第一阶段：保留现有系统，增加迭代节点
    • 第二阶段：逐步替换关键组件
    • 第三阶段：完整重构评估引擎

  2. 渐进式功能

    • 先实现迭代式生成，再扩展任务图
    • 评估系统最后升级，确保质量

  3. MVP 优先

    • 快速验证核心价值：多轮迭代是否真的改进质量
    • 收集真实用户反馈：基于实际使用调整系统
