LLM-as-a-Judge 评估系统

LLM-as-a-Judge 评估系统

更新日期：2026-03-12 来源：docs/work_logs/EVALUATION_LLM_JUDGE_PLAN.md

背景与目标

当前链路里，Sub Agent 运行代码后：

  1. 失败时返回报错。
  2. 成功时直接结束。

问题： success=true 只代表"代码执行成功"，不代表"模型形状符合需求"。

本阶段目标：

  1. 在现有执行链路后新增一级"建模质量反馈"。
  2. 先做成可扩展、可调参、可迭代的评估架构。
  3. 不追求一次性做出学术级严谨评分。

设计原则

  1. 推理与评分解耦：先固定证据，再评分。
  2. 评分规格显式化：rubric/version/weights 必须配置化。
  3. 评分器可插拔：规则评分器与 LLM 评分器并存。
  4. 失败可降级：评估失败不影响主执行语义。

总体架构

Sandbox execution result
    ↓
{执行状态检查}
    ├─ failure → 保持原返回
    └─ success → Evidence Builder
                    ↓
            ├─ Rule Evaluator
            └─ LLM Evaluator
                    ↓
                Score Merger
                    ↓
            ExecutionEvaluation (mode=llm_judge)

组件设计

1. Evidence Builder（统一评分输入）

职责： 把评分所需信息聚合成稳定结构，避免 prompt 漫游。

Evidence 字段：

字段说明
request_context用户需求文本、可选约束
execution_contextsuccess/stdout/stderr/error_code/timeout
artifact_context是否有 STEP、文件名、大小、预览信息
code_context生成代码文本（可截断）
metarubric_version/prompt_version/judge_model/evaluator_version
traceexecution_id/evaluation_trace_id/replay_key

2. Rule Evaluator（轻量确定性规则）

作用： 做硬性或半硬性信号，减少 LLM 误判面。

v1 规则：

  1. 无 STEP 产物：直接低分或 skipped
  2. 执行失败：不触发 LLM 评分
  3. 产物异常（空文件）：标记 critical_issue

注意： 规则层不是为了"替代 LLM"，是为了提供兜底和先验信号。

3. LLM Evaluator

六步流程：

1. 读取 rubric 配置 (维度、权重、锚点、输出 schema)
    ↓
2. 用 evidence 渲染评分 prompt
    ↓
3. 调用 judge model，要求返回 JSON
    ↓
4. 做 schema 校验
    ↓
5. 校验失败则单次重试（带 schema_fix 提示）
    ↓
6. 产出结构化分数与解释

输出 Schema（v1）

{
  "overall_semantic_score": 0.0,
  "confidence": 0.0,
  "dimension_scores": {
    "requirement_fidelity": 0.0,
    "geometric_reasonableness": 0.0,
    "manufacturability_proxy": 0.0,
    "code_quality_proxy": 0.0,
    "execution_stability": 0.0
  },
  "major_issues": ["..."],
  "suggestions": ["..."],
  "reasoning_brief": "..."
}

约束：

  1. 所有分数都在 [0,1]
  2. major_issues 与 suggestions 长度可控（最多 3 条）
  3. 不接受额外未知字段

Prompt 结构（模板化）

固定三段：

  1. 角色与任务：你是 CAD 评估器，不是生成器。
  2. Rubric 与锚点：逐维度给 1.0/0.5/0.0 的判定标准。
  3. 输出要求：只输出 JSON，遵守 schema。

4. Rubric 配置化

配置文件： configs/eval/cad_llm_judge_v1.yaml

配置示例：

rubric_id: cad_llm_judge
rubric_version: v1
prompt_version: cad_judge_prompt_v1

judge_model: claude-sonnet-4-20250514

weights:
  requirement_fidelity: 0.35
  geometric_reasonableness: 0.25
  manufacturability_proxy: 0.20
  code_quality_proxy: 0.10
  execution_stability: 0.10

anchors:
  requirement_fidelity:
    high: "关键尺寸与结构要点基本满足"
    mid: "主形状接近，但关键细节存在明显偏差"
    low: "与需求核心结构不一致"
  geometric_reasonableness:
    high: "拓扑与形体表达自然，无明显异常"
    mid: "局部有可疑结构或比例不协调"
    low: "明显几何异常或形体不合理"

hard_gates:
  missing_step_max_score: 0.20
  empty_artifact_max_score: 0.30

confidence:
  low_threshold: 0.65
  enable_rejudge: false

收益：

  1. 调权重不改代码
  2. 换模型不改代码
  3. 版本可追溯，可做灰度对比

5. Score Merger（融合与兜底）

职责：

  1. 合并 RuleEvaluator 与 LLMEvaluator 结果
  2. 应用 hard_gates（例如无 STEP 时封顶分）
  3. 映射为统一 ExecutionEvaluation

v1 融合建议：

  1. 有 LLM 结果时：final_score = llm_semantic_score
  2. 规则只做封顶和标记，不做复杂加权

可回放可比对系统

我们要回放什么

一次 llm_judge 至少要能还原 5 个对象：

  1. evidence_snapshot：评分输入快照
  2. rubric_snapshot：当时使用的 rubric 配置
  3. prompt_snapshot：真正发给 judge 的 prompt（脱敏后）
  4. judge_io：LLM 原始输出 + 解析输出 + 校验错误
  5. merged_result：最终写入 ExecutionEvaluation 的结果

最小追踪字段

每次评分都带以下标识：

字段说明
evaluation_trace_id本次评分唯一 id
execution_id对应执行链路 id
replay_key用于"同输入对比"
rubric_versionRubric 版本
prompt_versionPrompt 版本
judge_modelJudge 模型具体版本
evaluator_version评分代码版本

回放产物目录

eval_artifacts/{evaluation_trace_id}/
  evidence.json
  rubric.yaml
  prompt.txt
  judge_raw.json
  judge_parsed.json
  merge_result.json
  meta.json

输出给产品/前端的建议形态

推荐固定结构：

  1. score：0-1 分值（便于排序与看板）
  2. summary：一句话总评
  3. details.major_issues：问题清单
  4. details.suggestions：下一步建议
  5. details.rubric_version/prompt_version/judge_model：可追溯元数据
  6. details.evaluation_trace_id/replay_key/evaluator_version：可回放入口

实施顺序

Step 1（先跑通）

  • 实现 evaluation_orchestrator + evidence_builder + llm_judge
  • 固定一版 rubric yaml
  • 完成 schema 校验与失败兜底

Step 2（增强可调）

  • 接入 rule_evaluator + score_merger
  • 接入配置开关与版本字段
  • 支持一键切换 judge model 与 rubric

Step 3（后续增强）

  • 接入几何分析证据（可选）
  • 低置信度二评
  • 逐步提高分数稳定性
