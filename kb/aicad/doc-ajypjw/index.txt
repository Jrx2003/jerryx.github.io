### 内联脚本版本（备选）

```bash
cd /Users/jerryx/code/aicad.app.agent
set -a
source .env
set +a
export SANDBOX_MCP_BENCHMARK_INDEX_PATH=/Users/jerryx/code/aicad.app.agent/.tmp/sampled_10_per_L_extract/benchmark_index.json
export SANDBOX_MCP_LLM_JUDGE_ENABLED=false

uv run python - <<'PY'
import asyncio
import csv
import json
import os
from pathlib import Path
from mcp.client.session import ClientSession
from mcp.client.stdio import StdioServerParameters, stdio_client

repo = Path('/Users/jerryx/code/aicad.app.agent')
root = repo / '.tmp/sampled_10_per_L_extract/sampled_10_per_L'
out_path = repo / '.tmp/sampled_10_per_L_extract/results/mcp_ground_truth_eval_l1_l2.json'
out_path.parent.mkdir(parents=True, exist_ok=True)

rows = []
for lv in ['L1', 'L2']:
    csv_path = root / lv / f'{lv}_sampled_rows.csv'
    with csv_path.open('r', encoding='gbk', newline='') as f:
        reader = csv.DictReader(f)
        for row in reader:
            rows.append({
                'level': lv,
                'id': row['id'],
                'code': row['generated_code'],
                'requirement_text': row['geo_prompt_en'],
            })

params = StdioServerParameters(
    command='uv',
    args=['run', 'python', '-m', 'sandbox_mcp_server'],
    cwd=str(repo),
    env=os.environ.copy(),
)

async def main() -> None:
    results = []
    async with stdio_client(params) as (read_stream, write_stream):
        async with ClientSession(read_stream, write_stream) as session:
            await session.initialize()
            for row in rows:
                call = await session.call_tool('execute_cadquery', {
                    'code': row['code'],
                    'timeout_seconds': 120,
                    'include_artifact_content': False,
                    'benchmark_sample_id': row['id'],
                    'requirement_text': row['requirement_text'],
                })
                sc = call.structuredContent if isinstance(call.structuredContent, dict) else {}
                ev = sc.get('evaluation', {}) if isinstance(sc.get('evaluation'), dict) else {}
                results.append({
                    'level': row['level'],
                    'id': row['id'],
                    'success': sc.get('success'),
                    'error_code': sc.get('error_code'),
                    'eval_status': ev.get('status'),
                    'eval_score': ev.get('score'),
                    'eval_passed': ev.get('passed'),
                })

    total = len(results)
    success = sum(1 for r in results if r['success'] is True)
    payload = {
        'summary': {
            'total': total,
            'success_exec': success,
            'failed_exec': total - success,
        },
        'results': results,
    }
    out_path.write_text(json.dumps(payload, ensure_ascii=False, indent=2), encoding='utf-8')
    print('TOTAL', total, 'SUCCESS', success)
    print('REPORT', out_path)

asyncio.run(main())
PY
```

## 兼容性问题与修复

### 问题 1：cadquery-runtime 版本兼容

**现象：** `cadquery==2.4.0` 下，部分脚本中 `workplaneFromTagged("base")` 会返回空对象上下文，随后 `.faces(">Z")` 报错：

```
ValueError: Can not return Nth element of an empty list
```

**修复：** 在 sandbox 运行时加入兼容包装
- `workplaneFromTagged` 结果为空时，优先回退到 `ctx.tags[name]`（同名 tag 对象）
- 保留 `show_object -> result` 回填能力

---

### docker_runner.py 改动对比

| 改动点 | 改动前 | 改动后 | 改动原因 |
|---------|---------|---------|---------|
| 代码写入方式 | 直接把用户代码写入 `code.py` 并执行 | 新增 `_build_runtime_code(user_code)`，先注入 prelude/epilogue，再写入执行 | 数据集里的 CadQuery 代码风格不统一，直接执行会因运行时假设差异失败 |
| `cadquery as cq` | 不保证存在 | prelude 固定注入 `import cadquery as cq` | 部分样本默认 `cq` 已存在；在 sandbox 里未必成立 |
| `show_object/debug` | 无 | 注入 `show_object(obj, *args, **kwargs)` 与 `debug(*args, **kwargs)` 空实现 | 一些脚本用 `show_object` 导出结果而非 `result = ...`，缺失会报错或无输出 |
| `result` 兜底 | 无 | 若全局无 `result` 且存在 `show_object` 最后对象，则自动回填 `result` | 兼容 CQ-Editor 风格脚本，避免"执行成功但无 STEP" |
| `workplaneFromTagged` | 使用 CadQuery 原生行为 | monkey patch：先调原方法；若返回空对象，优先回退 `ctx.tags[name]`，否则回退 `self`；异常也回退 `self` | 在 `cadquery==2.4.0` 下，部分样本会出现 tagged workplane 为空 |

---

### 影响与边界

- **影响范围：** 仅 sandbox 执行容器内的临时代码，不改用户原始代码文件
- **正向效果：** L1/L2 的 20 条样本可稳定执行（`success_exec=20/20`）
- **代价：** 运行时更"宽容"，可能掩盖少量严格语义错误；因此此兼容层定位为"数据集/跨环境适配层"，而非 CAD 语义修正器

                    'eval_status': ev.get('status'),
                    'eval_score': ev.get('score'),
                    'eval_passed': ev.get('passed'),
                })

    total = len(results)
    success = sum(1 for r in results if r['success'] is True)
    payload = {
        'summary': {
            'total': total,
            'success_exec': success,
            'failed_exec': total - success,
        },
        'results': results,
    }
    out_path.write_text(json.dumps(payload, ensure_ascii=False, indent=2), encoding='utf-8')
    print('TOTAL', total, 'SUCCESS', success)
    print('REPORT', out_path)

asyncio.run(main())

## 兼容性问题与修复

### 问题 1：cadquery-runtime 版本兼容

**现象：** `cadquery==2.4.0` 下，部分脚本中 `workplaneFromTagged("base")` 会返回空对象上下文，随后 `.faces(">Z")` 报错：

```
ValueError: Can not return Nth element of an empty list
```

**修复：** 在 sandbox 运行时加入兼容包装
- `workplaneFromTagged` 结果为空时，优先回退到 `ctx.tags[name]`（同名 tag 对象）
- 保留 `show_object -> result` 回填能力

---

### docker_runner.py 改动对比

| 改动点 | 改动前 | 改动后 | 改动原因 |
|---------|---------|---------|---------|
| 代码写入方式 | 直接把用户代码写入 `code.py` 并执行 | 新增 `_build_runtime_code(user_code)`，先注入 prelude/epilogue，再写入执行 | 数据集里的 CadQuery 代码风格不统一，直接执行会因运行时假设差异失败 |
| `cadquery as cq` | 不保证存在 | prelude 固定注入 `import cadquery as cq` | 部分样本默认 `cq` 已存在；在 sandbox 里未必成立 |
| `show_object/debug` | 无 | 注入 `show_object(obj, *args, **kwargs)` 与 `debug(*args, **kwargs)` 空实现 | 一些脚本用 `show_object` 导出结果而非 `result = ...`，缺失会报错或无输出 |
| `result` 兜底 | 无 | 若全局无 `result` 且存在 `show_object` 最后对象，则自动回填 `result` | 兼容 CQ-Editor 风格脚本，避免"执行成功但无 STEP" |
| `workplaneFromTagged` | 使用 CadQuery 原生行为 | monkey patch：先调原方法；若返回空对象，优先回退 `ctx.tags[name]`，否则回退 `self`；异常也回退 `self` | 在 `cadquery==2.4.0` 下，部分样本会出现 tagged workplane 为空 |

---

### 影响与边界

- **影响范围：** 仅 sandbox 执行容器内的临时代码，不改用户原始代码文件
- **正向效果：** L1/L2 的 20 条样本可稳定执行（`success_exec=20/20`）
- **代价：** 运行时更"宽容"，可能掩盖少量严格语义错误；因此此兼容层定位为"数据集/跨环境适配层"，而非 CAD 语义修正器
