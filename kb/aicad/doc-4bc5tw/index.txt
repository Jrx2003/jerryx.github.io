c:T1d38,# 测试产物目录结构

> 更新日期：2026-03-12
> 来源：`docs/work_logs/ACI_终端运行与产物查看指南_20260309.md`

## 概述

测试产物目录用于保存测试执行过程中的所有中间结果，便于调试、分析和验证。

---

## 运行根目录

```
.tmp/test_runs/<YYYYMMDD_HHMM>/
```

例如：`.tmp/test_runs/20260309_2340/`

---

## 完整目录结构

```
<run>/
│
├── run_manifest.json              # 本次运行参数、命令、provider/model、日志路径总览
│
├── logs/
│   ├── unit_aci.log             # 单元测试原始输出
│   └── e2e_real_aci.log         # 真实 LLM + sandbox 执行日志
│
└── cases/
    ├── comparison_summary.json    # 每个 case 的迭代式与一次性总结果
    │
    ├── case_01_plate_fillet/
    │   ├── input.json            # 测试用例输入
    │   │
    │   ├── iterative/
    │   │   ├── prompts/
    │   │   │   ├── round_01_request.json      # 每轮拼装后的提示词
    │   │   │   ├── round_02_request.json
    │   │   │   └── ...
    │   │   │
    │   │   ├── plans/
    │   │   │   ├── round_01_response.json     # 每轮 LLM 返回动作计划
    │   │   │   ├── round_02_response.json
    │   │   │   ├── round_01_llm_retry_01.json  # LLM 重试记录
    │   │   │   └── ...
    │   │   │
    │   │   ├── actions/
    │   │   │   ├── round_01_action_01_request.json  # 实际送入的动作序列
    │   │   │   ├── round_01_action_01_result.json   # 每一步执行结果
    │   │   │   └── ...
    │   │   │
    │   │   └── outputs/
    │   │       ├── step_after_action_01.step  # 每一步的 STEP 快照
    │   │       ├── step_after_action_02.step
```

## 使用场景

### 场景 1：调试 LLM 推理

查看 `prompts/round_XX_request.json` 和 `plans/round_XX_response.json`

1. 理解 LLM 收到的提示词
2. 分析 LLM 返回的动作计划
3. 检查是否理解了需求

---

### 场景 2：追踪执行过程

查看 `actions/` 目录中的所有结果

1. 按顺序查看每个动作的执行结果
2. 分析每步的 `snapshot` 变化
3. 检查 `suggestions` 和 `completeness`

---

### 场景 3：对比迭代式 vs 一次性

打开同一个 case 的两个 `final_model.step`

1. 使用 CAD 软件打开对比
2. 检查几何差异
3. 分析哪种方式更符合需求

---

### 场景 4：分析错误

查看 `e2e_real_aci.log` 和失败的 `*_result.json`

1. 找到失败的动作
2. 查看 `error` 字段
3. 分析失败原因并修复

---

### 场景 5：性能分析

查看 `summary.json` 和所有 `*_response.json`

1. 统计总轮数和总动作数
2. 分析 token 使用情况
3. 计算效率指标

---

## 目录大小估算

单个 case 的产物大小：
- prompts: ~10KB (每个 request ~2KB)
- plans: ~5KB (每个 response ~1KB)
- actions: ~50KB (每个 result ~5KB，10 个动作)
- outputs: ~100KB (STEP 文件 + 截图)
- summary: ~2KB

**总计：** ~170KB / case

3 个 case：~510KB

## 使用场景

### 场景 1：调试 LLM 推理

查看 `prompts/round_XX_request.json` 和 `plans/round_XX_response.json`

1. 理解 LLM 收到的提示词
2. 分析 LLM 返回的动作计划
3. 检查是否理解了需求

---

### 场景 2：追踪执行过程

查看 `actions/` 目录中的所有结果

1. 按顺序查看每个动作的执行结果
2. 分析每步的 `snapshot` 变化
3. 检查 `suggestions` 和 `completeness`

---

### 场景 3：对比迭代式 vs 一次性

打开同一个 case 的两个 `final_model.step`

1. 使用 CAD 软件打开对比
2. 检查几何差异
3. 分析哪种方式更符合需求

---

### 场景 4：分析错误

查看 `e2e_real_aci.log` 和失败的 `*_result.json`

1. 找到失败的动作
2. 查看 `error` 字段
3. 分析失败原因并修复

---

### 场景 5：性能分析

查看 `summary.json` 和所有 `*_response.json`

1. 统计总轮数和总动作数
2. 分析 token 使用情况
3. 计算效率指标

---

## 目录大小估算

单个 case 的产物大小：
- prompts: ~10KB (每个 request ~2KB)
- plans: ~5KB (每个 response ~1KB)
- actions: ~50KB (每个 result ~5KB，10 个动作)
- outputs: ~100KB (STEP 文件 + 截图)
- summary: ~2KB

**总计：** ~170KB / case

3 个 case：~510KB
